Below is a product spec + file format spec + API docs for what we’ve described: an OBJ-like 3D layer where beliefs are vertices, triangles encode local coherence, and stretch/strain dampens confidence—packaged as a single human-readable file that opens into a Postman-like browser UI for agents.

I’m going to name the ecosystem for clarity:
	•	File type: .sem (Semantics Mesh)
	•	Desktop/UI app: Semantics Studio
	•	Local runtime/API: Semantics Runtime

You can rename later without changing the design.

⸻

Semantics Mesh v0.1 Product Spec

Product promise

A .sem file is a portable semantic workspace:
	•	Human-readable: open in any text editor
	•	Machine/agent-readable: structured records + stable IDs
	•	Visually inspectable: double-click opens a browser UI showing a 3D mesh
	•	Agent-agnostic: any agent can read/write via a simple local API
	•	Coherence-aware: “belief confidence” is dampened by geometric strain in triangles

Mental model
	•	A belief is a vertex in 3D space.
	•	A triangle ties three beliefs together as a minimal “closed” coherence unit.
	•	When beliefs move, triangles stretch.
	•	Stretch produces strain.
	•	Strain reduces effective confidence (and can optionally trigger “suggested repairs”).

This is a semantic interface and coherence simulator, not a “cognitive simulation.”

⸻

File Format Spec: .sem (OBJ-compatible + JSON records)

Design goals
	1.	Keep the OBJ 3D layer: vertices and faces are first-class.
	2.	Allow multiple “types” in one file (beliefs, triangles, metadata, events, requests).
	3.	Remain diff-friendly for Git.
	4.	Be safely parseable by a runtime without executing code.

High-level structure

A .sem file is:
	•	A valid Wavefront OBJ geometry stream: v (vertices), f (faces).
	•	Plus structured JSON records embedded as comments using a reserved prefix: #@ 

Why comments?
	•	OBJ tooling ignores comments, but still loads the geometry.
	•	Semantics Runtime reads comments as one JSON object per line (JSONL-style).

Reserved comment prefix
	•	#@  means “Semantics record (JSON object follows)”
	•	Other comments # ... are freeform and ignored by the runtime.

⸻

Canonical mapping rules

OBJ vertices and faces are positional lists; semantics need stable IDs.

Vertex ↔ belief mapping

Each belief record binds to a vertex index:
	•	OBJ vertex indices are 1-based (standard OBJ).
	•	A belief record must include "vertex": <int>.

Face ↔ triangle mapping

Each triangle record binds to a face index, or explicitly lists vertex indices:
	•	"face": <int> (1-based in the file’s face order), or
	•	"vertices": [v1, v2, v3]

⸻

Required header record

The file must include a format header near the top:

#@ {"type":"header","sem_version":"0.1","space":"R3","units":"arb","created_at":"2026-02-20T00:00:00Z"}

	•	space is "R3" for this v0.1 spec (your “must keep 3 dimensions” constraint).

Core record types

1) Belief record (type: "belief")

Represents the semantic payload attached to a vertex.

Schema (v0.1):

{
  "type": "belief",
  "id": "b_...",                 // stable string ID (UUID, hash, etc.)
  "vertex": 1,                   // 1-based OBJ vertex index
  "proposition": "…",            // human-readable belief statement
  "confidence_base": 0.0,        // 0..1, prior confidence before strain
  "tags": ["optional", "tags"],
  "provenance": {
    "source": "human|agent|tool",
    "author": "optional",
    "refs": ["optional external refs"]
  },
  "embedding": {
    "model": "optional",
    "dims": 1536,
    "codec": "float32|base64",
    "data": "optional in-file or omitted"
  },
  "updated_at": "ISO-8601"
}

Notes:
	•	Position is not stored in JSON; it comes from the OBJ v x y z line at that vertex index.
	•	Embedding payload is optional. For v0, you can omit it entirely and still have a working system.

2) Triangle record (type: "triangle")

Defines a coherence simplex between 3 beliefs.

Schema (v0.1):

{
  "type": "triangle",
  "id": "t_...",
  "face": 1,                     // or omit and use "vertices":[]
  "vertices": [1,2,3],            // optional explicit mapping
  "rest": {
    "edge_lengths": [1.0, 1.0, 1.0] // L12, L23, L31 (rest/ideal lengths)
  },
  "physics": {
    "stiffness": 1.0,             // spring strength
    "damping": 0.2                // local damping
  },
  "semantics": {
    "relation": "coherence",
    "notes": "optional"
  },
  "updated_at": "ISO-8601"
}

Notes:
	•	A triangle corresponds to an OBJ face f i j k.
	•	Rest lengths can be:
	•	explicit, or
	•	computed at creation time from current geometry (then persisted).

3) Edge record (type: "edge") — optional in v0.1

For pairwise constraints, useful when triangles aren’t present.

{
  "type":"edge",
  "id":"e_...",
  "vertices":[1,2],
  "rest":{"length":1.0},
  "physics":{"stiffness":0.5,"damping":0.2},
  "updated_at":"ISO-8601"
}

4) Event record (type: "event") — for append-only edits

Events are how agents (or UI) mutate state while keeping Git diffs sane.

{
  "type":"event",
  "id":"ev_...",
  "ts":"ISO-8601",
  "actor":{"type":"human|agent","name":"…"},
  "op":"move_vertex|set_confidence|set_proposition|add_triangle|remove_triangle",
  "args": { }
}

Recommended: events are append-only; the runtime can “materialize” a snapshot on save.

5) Request record (type: "request") — the “Postman for agents” piece

This is how .sem becomes an executable, agent-facing workspace.

{
  "type":"request",
  "id":"r_...",
  "name":"Summarize local neighborhood",
  "input": {
    "select": {
      "mode":"neighborhood",
      "center_belief_id":"b_...",
      "radius": 2,
      "max_items": 50
    }
  },
  "agent": {
    "interface":"generic",
    "instruction":"Generate a concise summary of these beliefs and tensions.",
    "output_format":"patch_v0"
  },
  "apply": {
    "mode":"suggest",             // "suggest" | "auto" (auto requires policy)
    "target":"workspace"
  }
}

Coherence Engine Spec: “Semantic Physics” v0.1

This is intentionally small: it’s not rigid-body physics; it’s a local relaxation system.

State
	•	Vertex positions p_i ∈ R³ come from OBJ v lines.
	•	Triangles define constraints via rest edge lengths.

Strain / energy for a triangle

For triangle with vertices (a,b,c), define edge lengths:
	•	d_ab = ||p_a - p_b||
	•	d_bc = ||p_b - p_c||
	•	d_ca = ||p_c - p_a||

Rest lengths from triangle record:
	•	L_ab, L_bc, L_ca

Define normalized strain per edge:
	•	s_ab = (d_ab - L_ab) / max(L_ab, ε)  (ε small > 0)

Triangle strain energy (simple):
	•	E_tri = k * (s_ab² + s_bc² + s_ca²)
where k = stiffness.

Global energy is sum over triangles (and optional edges).

“Relaxation” update

When something changes (a vertex dragged, event applied):
	•	Only recompute forces in a local neighborhood (incident triangles).
	•	Perform N small steps of gradient descent / verlet-like iteration.

Parameters:
	•	dt step size (small)
	•	damping reduces oscillation

Confidence dampening (your rule)

Each belief has confidence_base in [0,1].

We compute effective confidence:
	•	Let strain_i = aggregate strain from triangles touching vertex i:
	•	e.g. average of incident triangle strains, clamped to ≥ 0

Then:
	•	confidence_effective = confidence_base * exp(-α * strain_i)

Where:
	•	α defaults to something like 1.0 (tunable)
	•	This creates a smooth dampening that increases as triangles stretch

Interpretation:
	•	Beliefs can move freely in 3D
	•	But strained neighborhoods become “less trusted” without deleting anything

Semantics Product Spec

1. Summary

Semantics (.sem) is a single-file, human-editable, machine-readable, agent-readable semantic interface that stores a 3D belief mesh and opens into a browser-based UI (Postman-for-agents style) for:
	•	inspecting beliefs as points in a 3D mesh
	•	viewing relationships as triangles
	•	querying the mesh (search, neighbors, hotspots)
	•	mutating beliefs (move points, add triangles, retract)
	•	running a coherence relaxation step that computes strain and dampens confidence

It is not a general cognition engine. It’s a persistent semantic workspace.

⸻

2. Core primitives

Belief (primitive)

A belief is the smallest atomic unit. In .sem, each belief maps to:
	•	a vertex in 3D: (x, y, z)
	•	a proposition (text)
	•	a base confidence
	•	derived metrics computed by the coherence engine (strain, derived confidence)

Triangle (minimal coherence unit)

A triangle represents a 3‑way coherence relation among three beliefs.
	•	triangles come from mesh faces (f a b c)
	•	triangle “stretch” represents tension among those beliefs
	•	triangle stretch contributes to confidence dampening on its vertices

Change

A change can be expressed as:
	•	direct edits to geometry (v lines) and metadata (#@{...})
	•	optional append-only events (#@{type:"event", ...}) for auditability

⸻

3. Product goals

Goals
	•	One file that can be checked into git and diffed.
	•	Open in text editor and understand/modify it.
	•	Double-click opens browser UI (via Semantics Studio runtime).
	•	Agent-agnostic read/write contract.
	•	OBJ-like 3D layer is real, not metaphor.
	•	Coherence simulation based on geometric strain; dampen confidence when stretched.

Non-goals
	•	Not a full reasoning system.
	•	Not training data format.
	•	Not a “mind” or cognitive simulator.
	•	Not a universal truth oracle.

Semantics Studio UI (browser) must include:
	•	3D Mesh View: beliefs as points; triangles as faces; tension heatmap
	•	Inspector: click a belief → show proposition, base confidence, derived confidence, provenance

Semantics File Format Spec

1. File type
	•	Extension: .sem (alias: .semantics)
	•	MIME (suggested): application/x-semantics+obj
	•	Encoding: UTF‑8
	•	Line-oriented
	•	Valid OBJ superset: any normal .obj parser can ignore semantic lines.

Design trick

All semantic records are stored as OBJ comments that begin with #@:
	•	OBJ ignores them (# ... is a comment)
	•	Humans can edit them
	•	Each one is a single JSON object (JSONL-style)

So the file is simultaneously:
	•	OBJ geometry (v, f, …)
	•	JSONL overlay (#@{...} one object per line)

⸻

2. Grammar overview

Geometry layer (OBJ subset)

Required:
	•	v x y z — vertex positions (belief points)
	•	f i j k — triangle faces (coherence triangles), 1-based indices

Optional (supported but not required):
	•	o name, g name — grouping
	•	usemtl, etc. (ignored by semantics unless referenced)

Overlay layer (Semantics JSONL)

Lines of the form:

#@{ ...valid JSON object... }


⸻

3. Canonical indexing rules
	•	Vertex indices are OBJ vertex indices (1-based).
	•	Face indices are the order of f lines (1-based), unless you store explicit vertex triplets.

A belief record references either:
	•	v: vertex index (recommended)
	•	id: stable identifier (required)

A triangle record references either:
	•	f: face index (recommended)
	•	verts: explicit [i, j, k]

⸻

4. Required overlay record types

4.1 Schema record (required once)

Defines versioning and defaults.

#@{"type":"schema","version":"0.1","id":"mesh:demo","confidence_model":"strain_dampen_v1"}

Fields
	•	type: "schema"
	•	version: semantic format version
	•	id: mesh identifier
	•	confidence_model: "strain_dampen_v1" (v0 default)

Optional:
	•	units, coordinate_system, created_at, authors, tags

⸻

4.2 Belief record (required per belief)

Maps a belief to a vertex.

#@{"type":"belief","id":"b:sky-blue","v":1,"p":"The sky is blue.","c":0.92,"tags":["perception"]}

Fields
	•	type: "belief"
	•	id: stable belief id (string)
	•	v: vertex index (int, 1-based)
	•	p: proposition text (string)
	•	c: base confidence (0..1)

Optional:
	•	status: "active" | "retracted"
	•	source: provenance object (see below)
	•	updated_at: ISO8601
	•	locked: boolean (prevents relaxation from moving it unless explicitly moved)
	•	mass: float (default 1.0)
	•	embedding: { "dim": 1536, "ref": "embeddings.bin#offset=..." } (optional)

⸻

4.3 Triangle record (required per coherence triangle)

Binds semantics to a face and defines “rest” shape for strain.

#@{"type":"triangle","id":"t:sky","f":1,"rest":{"mode":"from_current"},"k":1.0}

Fields
	•	type: "triangle"
	•	id: stable triangle id
	•	f: face index (1-based) OR verts: [i,j,k]
	•	rest: rest-shape definition
	•	{ "mode":"from_current" } means: rest lengths are captured from current vertex positions at load-time (or on first creation)
	•	or { "edge_lengths":[L12,L23,L31] } for explicit rest lengths
	•	k: stiffness (spring constant) default 1.0

Optional:
	•	tags
	•	notes
	•	created_at

⸻

5. Optional overlay record types

5.1 Evidence

Attach external evidence to beliefs (provenance, citations, excerpts).

#@{"type":"evidence","id":"e:1","belief":"b:sky-blue","weight":0.7,"source":{"kind":"url","ref":"..."},"excerpt":"..."}

5.2 Request (Postman-like)

Saved, runnable operations.

#@{"type":"request","id":"r:hotspots","name":"Tension hotspots","method":"POST","path":"/v1/query","body":{"op":"hotspots","top":10}}

5.3 Environment variables

Postman-style env

#@{"type":"env","vars":{"TOP":10,"HOPS":2}}

5.4 Event log (append-only)

Optional audit trail, merge-friendly.

#@{"type":"event","ts":"2026-02-20T17:10:00Z","actor":"agent:alpha","op":"move","belief":"b:sky-blue","to":[0.2,0.1,0.0],"reason":"new evidence"}

v0 rule: if events exist, Semantics Studio may either (a) apply them on top of current v snapshot, or (b) treat the file as event-sourced depending on schema.mode.

Coherence Engine Spec

1. What it is (and isn’t)

It’s a local relaxation + strain computation system:
	•	Beliefs can be moved anywhere in 3D.
	•	Triangles “want” to keep their rest shape (edge lengths).
	•	Stretch → strain → confidence dampening.
	•	This yields a coherent interface and diagnostic for tension.

It is not:
	•	proof-based logic
	•	global reasoning
	•	“truth” computation

⸻

2. Data extracted from file

From OBJ:
	•	vertices: x_i ∈ ℝ³
	•	faces (triangles): (i,j,k)

From overlay:
	•	belief ↔ vertex mapping
	•	triangle stiffness k
	•	rest lengths L0 per triangle edge

⸻

3. Strain model

For each triangle with vertices (i, j, k) and rest edge lengths:
	•	L0_ij, L0_jk, L0_ki

Compute current edge lengths:
	•	L_ij = ||x_i - x_j||, etc.

Edge strain:
	•	s_ij = |L_ij - L0_ij| / (L0_ij + ε)

Triangle strain (default):
	•	S_tri = (s_ij² + s_jk² + s_ki²) / 3

Vertex local strain (for belief at vertex i):
	•	average triangle strain over incident triangles
	•	or average edge strain over incident edges

⸻

4. Confidence dampening

Beliefs store base confidence c ∈ [0,1].

Engine computes derived confidence c' (not necessarily stored in file; computed live):

Default dampening:
	•	c' = c · exp(-α · S_local)

Where:
	•	α is a global sensitivity (default e.g. 2.0)
	•	S_local is the belief’s local strain

Alternative (more linear):
	•	c' = c / (1 + β · S_local)

⸻

5. Relaxation / “physics” step

We do not need rigid body physics. We need spring relaxation.

For each triangle edge (a,b):
	•	define spring with rest length L0_ab
	•	stiffness derived from triangle k

Force direction:
	•	along (x_a - x_b)

Integrate with simple damped steps (semi-implicit Euler is fine for v0).

Constraints:
	•	3D positions always stay 3D
	•	locked nodes don’t move during relaxation
	•	optional mass per node affects movement

Outputs:
	•	updated positions (if you choose to write back)
	•	computed strain + derived confidences
	•	hotspots and contradictions candidates (high strain clusters)

Semantics Studio Runtime

1. What happens when you “click the file”

Semantics Studio (desktop app or CLI launcher) does:
	1.	Parse .sem file (OBJ + overlay)
	2.	Build in-memory graph (vertices ↔ beliefs ↔ triangles)
	3.	Start local server (localhost)
	4.	Open browser UI bound to that file
	5.	Any edits can be:
	•	live in UI (and saved to file)
	•	live in text editor (Studio auto-reloads)

⸻

2. “File acts like an API”

Two meanings:
	1.	The runtime exposes an HTTP API to interact with the file’s state.
	2.	The file itself can contain saved request definitions (#@{"type":"request",...}) which the UI can run.

So .sem becomes:
	•	state container
	•	query surface
	•	request collection

5. Queries (Postman-like)

	•	search { q, top }
	•	neighbors { belief, hops, top }
	•	nearest { pos:[x,y,z], radius, top }
	•	hotspots { top } (highest strain triangles/regions)
	•	contradictions { top } (heuristics: high tension + opposing tags/provenance conflict)
	•	submesh { belief, hops } (return induced local mesh)

Git + Scaling Notes

Why git fits naturally
	•	.sem is line-based.
	•	Semantic overlay is JSONL-like.
	•	You can diff changes in:
	•	propositions
	•	confidence values
	•	triangle definitions
	•	event lines
	•	You can treat modifications as “local forces”: moving one vertex causes local strain changes (visible + computable).

Scaling (practical v0 stance)

You don’t need infinite scale. You need:
	•	sparse triangles
	•	local neighborhoods
	•	incremental recomputation

If you do need more:
	•	build an index sidecar .sem.idx (vertex offsets, belief id → line offset)
	•	optionally store embeddings in a binary sidecar .sem.emb to avoid JSON bloat

What you asked for, delivered “exactly”
	•	✅ Still a file you can open in a text editor
	•	✅ OBJ 3D layer (v and f)
	•	✅ Belief primitive lives as a point (vertex) in triangles (faces)
	•	✅ Stretching triangles dampens confidence (strain → derived confidence)
	•	✅ Coherence simulator, not cognitive simulation
	•	✅ Browser UI like Postman for agents
